# 前端架构规范文档

## 1. 目标

本前端以 `assistant-ui` 为交互基座，以 `Next.js App Router` 为应用壳，以“页面层 / 组件层 / 应用层 / 接口层”四层结构组织代码，目标是：

- 层次清晰，页面代码不直接散落接口细节。
- SSR 与 CSR 共享统一业务语义，不重复维护接口协议。
- 管理台与聊天入口保持同一套设计规则，但分离各自职责。
- 在后续扩展文档中心、任务中心、聊天工具调用时，避免再次出现页面即接口、组件即业务的耦合。

## 2. 目标目录结构

```text
client/
├─ app/                         # 路由层，只做页面装配、首屏预取、布局组织
├─ components/
│  ├─ assistant-ui/            # assistant-ui 官方模板相关组件
│  ├─ console/                 # 管理台业务组件
│  └─ ui/                      # 通用 UI 原子组件
├─ lib/
│  ├─ api/                     # 统一接口层
│  │  ├─ compat-types.ts
│  │  ├─ compat-gateway.ts
│  │  ├─ browser-compat-transport.ts
│  │  └─ server-compat-transport.ts
│  ├─ chatApi.ts               # 聊天运行时 facade
│  ├─ langgraphApi.ts          # 浏览器侧管理台 facade
│  └─ serverCompatApi.ts       # SSR 侧 facade
└─ ...
```

## 3. 分层职责

### 3.1 路由层 `app/`

职责：

- 定义 URL 与页面装配关系。
- 做页面级服务端预取。
- 传递 `initialData` 给客户端组件。

禁止：

- 直接写复杂业务逻辑。
- 直接拼装多个 HTTP 协议细节。
- 在多个页面重复写相同的接口调用流程。

### 3.2 组件层 `components/`

职责：

- 负责页面呈现、交互状态、本地表单行为。
- 通过 props 接收初始数据与回调。
- 对用户动作进行最小范围的状态更新。

禁止：

- 组件内部定义接口协议字符串。
- 在多个组件中复制相同的请求拼装逻辑。
- 在 UI 组件中混入页面级数据装配。

### 3.3 应用接口层 `lib/*.ts`

职责：

- 作为“前端应用服务 facade”，对页面暴露清晰函数。
- 统一错误语义。
- 屏蔽 transport、协议字段名、兼容层路径。

要求：

- `langgraphApi.ts` 只面向浏览器侧。
- `serverCompatApi.ts` 只面向 SSR / Server Component。
- `chatApi.ts` 只面向 assistant-ui runtime。

### 3.4 传输层 `lib/api/`

职责：

- 统一封装 HTTP 请求、基础 URL、错误处理。
- 统一维护 compat API 的请求协议。
- 统一维护共享响应类型。

要求：

- 所有 `/compat/*` 路径都从这里收口。
- 不允许在页面组件中再次定义 `fetch('/compat/...')`。

## 4. 数据获取规范

### 4.1 管理台页面

默认采用：

- `Server Component` 首屏预取
- `Client Component` 本地交互刷新

适用页面：

- `/documents`
- `/documents/[documentId]`
- `/jobs/ingest`
- `/jobs/delete`

原因：

- 首屏稳定，不依赖浏览器 hydration 时机。
- 对自动化验收和弱网场景更稳。
- 与管理型列表页的 SEO/首屏可读性更匹配。

补充约束：

- 首屏注入的 `page contract` 不能只消费 `filters/sorts`。
- 列表页必须优先消费 contract 中的 `tables` 定义生成列结构。
- 详情页必须优先消费 contract 中的 `sections.fields`、`sections.stat_cards`、`sections.tables`，禁止页面重复硬编码同一组字段。
- 存在 `tabs` 时，页面应根据 `section_keys` 切换渲染内容，不再手写切换逻辑。
- 对于 `ingest/reindex/delete` 这类任务型动作，页面交互语义必须优先区分“任务已提交”和“任务已完成”。
- `documents` 页中的 URL/文件入库按钮只负责提交任务并返回 `job_id`，不得同步等待整个入库链执行完成后才反馈成功。
- 提交任务成功后，页面应提示用户前往 `/jobs/ingest` 查看进度，而不是把主列表区长期维持在阻塞式加载态。
- `documents` 页的 URL 入库应支持站内递归抓取配置；若启用递归，表单至少要显式暴露开关与最大深度，并默认限制在当前 URL 范围内。
- 递归 URL 入库后的数据展示语义必须保持“每个子页面是独立文档”，禁止在前端把整站抓取结果误当成单一页面文档。
- 管理台的任务列表、文档列表和详情页，在检测到 `created/pending/processing/running` 等活跃状态时，应进行静默轮询刷新，避免用户必须手动刷新才能看到生命周期推进。
- 表格型列表应优先保证可读性：宁可提供横向滚动和合理最小列宽，也不要把表头或长文本挤压成过多换行导致信息难以扫描。
- console 与 assistant 两类工作区侧栏都应支持用户主动收缩，并持久化收缩状态，避免固定宽侧栏长期占用主内容区域。

### 4.2 聊天页面

默认采用：

- 纯客户端运行时
- 由 `assistant-ui` runtime 驱动
- 使用专门的聊天 facade，不与管理台请求共用页面级状态逻辑
- 聊天页允许在 `assistant-ui` 外层增加“会话列表”这类工作区外壳，但步骤轨迹必须跟随具体消息轮次渲染，不能脱离消息流放在固定总控面板中

补充约束：

- 聊天发送必须优先走流式接口，不能退回到一次性整包返回。
- 若聊天 runtime 基于 `useLangGraphRuntime`，前端流式适配层必须透传兼容的 LangGraph `messages` tuple 事件，不能只依赖最终 `values` 快照，否则回答会表现为非流式回填。
- 若历史消息或最终 `values` 消息需要在 `assistant-ui` 中显示自定义轨迹信息，前端归一化时必须把这类数据并入 `additional_kwargs.metadata.custom`，不能只挂在 `response_metadata` 或自定义顶层字段。
- 聊天运行中的分析、检索、生成步骤必须按“当前提问 -> 当前回答”的粒度内嵌到消息区域。
- 实时运行阶段应优先显示在当前用户问题下方；当本轮回答落库后，再由助手消息中的持久化轨迹承接展示。
- 问题分析阶段应在当前问题下方显示运行态，不得等回答完成后才统一回填。
- 知识检索阶段应至少展示检索摘要，并支持展开查看命中的知识片段列表。
- 历史消息回放时，必须能从助手消息元数据恢复同一轮次的步骤轨迹，而不是只在实时流中可见。
- 会话列表中的新线程创建必须只有一条前端入口，优先由线程列表初始化流程负责；运行时层不得再并行创建第二条远程线程。
- 会话标题必须优先取用户消息文本，不能用整段助手回答或整轮消息拼接结果作为标题。
- 会话历史不能只存在浏览器内存，必须有真实持久化来源。

## 5. 命名规范

- 页面组件：`DocumentsPage`、`JobsPage`
- 路由页：`DocumentsRoutePage`、`IngestJobsRoutePage`
- 业务 facade：`getPageContract()`、`listDocuments()`
- transport：`requestBrowserCompat()`、`postServerCompat()`

要求：

- 名称必须体现所在层级和职责。
- 避免使用 `utils`、`helper` 这类宽泛命名承载业务能力。

## 6. 依赖方向

允许：

- `app -> components`
- `app -> lib`
- `components -> lib`
- `lib facade -> lib/api`

禁止：

- `lib/api -> components`
- `components/ui -> components/console`
- `components -> app`

## 7. 错误处理规范

- 浏览器侧 facade 失败时抛异常，由页面状态统一承接。
- 服务端 facade 失败时返回带 `error` 的对象，保证页面首屏可降级渲染。
- 页面必须显式区分：
  - `loading`
  - `error`
  - `empty`
  - `ready`

## 8. 扩展规范

新增一个管理页时，顺序必须为：

1. 后端先定义 page contract 与 read-model operation。
2. 前端在 `lib/api` 中补 gateway 调用。
3. 路由页决定是否需要 SSR 首屏预取。
4. 页面组件只实现视图与交互，不重新定义接口协议。

补充要求：

- 新增或修改管理页时，优先更新 `client/lib/contracts/page-contract.ts` 的解析能力，而不是在页面组件内临时拼装字段路径。
- 页面中的格式化逻辑只负责把 contract 值映射为 UI 表现，例如 badge、datetime、link；字段集合本身必须来自后端 contract。

## 9. 当前阶段重构结论

本次已完成的第一阶段收敛：

- 抽出统一 `compat-types / transport / gateway`
- 将浏览器侧与服务端侧 compat 调用语义统一
- 将列表页改为首屏预取，消除“页面一直 loading”的不稳定行为

后续建议继续推进：

- 将 page contract 从“部分消费”升级为“明确消费边界”，避免再次出现双维护

本次已完成的第三阶段收敛：

- `document detail` 迁移到 `Server Component` 首屏预取 + `Client Component` 本地刷新
- `jobs/delete` 迁移到与 `jobs/ingest` 一致的数据策略
- 管理台四个核心页面现在统一为同一套首屏数据策略：
  - `/documents`
  - `/documents/[documentId]`
  - `/jobs/ingest`
  - `/jobs/delete`

本次已完成的第四阶段收敛：

- 新增 `client/lib/contracts/page-contract.ts`，统一解析 `filters/sorts/tables/sections/stat_cards`
- `/documents` 已改为按 `document_table` contract 渲染列表列
- `/documents/[documentId]` 已改为按 `summary/stats/recent_versions/versions_table` contract 渲染
- `/jobs/ingest` 与 `/jobs/delete` 已改为按 contract 渲染任务列表、任务摘要、统计卡及关联数据块

本次已完成的聊天链路收敛：

- `chatApi.ts` 已切换到 `/compat/assistant/chat/stream` 的流式 NDJSON 协议
- 聊天页已增加会话列表，并将“问题分析 / 知识检索 / 答案生成”收敛为每轮消息内联轨迹
- 助手消息已补充检索摘要、可展开检索命中列表与知识来源展示
- 当前线程 ID 已在前端保留，并与后端持久化线程列表配合
