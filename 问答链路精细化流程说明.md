# 问答链路精细化流程说明

本文档专门说明当前项目中“用户提问之后，系统到底怎么处理”的完整链路，重点回答这些问题：

- 用户消息从哪里进入后端
- 后端先做什么判断
- 是否做召回
- 用什么模型做分析、召回、重排、生成
- 检索结果如何返回给前端
- 最终答案和轨迹如何持久化

本文档不展开文档入库、删除、重建等后台管理流程，主要聚焦聊天问答主链路。

## 1. 一句话概括

当前项目的问答流程本质上是：

`会话线程恢复 -> 最近对话记忆判断 -> 问题分析 -> 查询改写 -> 向量召回 -> 可选重排 -> 大模型生成 -> 流式返回 -> 持久化轨迹`

也就是说，这不是一个“用户一问，大模型直接答”的纯聊天系统，而是一个带有知识库召回能力、会话历史记忆、流式输出和轨迹回放的 RAG 问答链路。

## 2. 主入口在哪里

前端聊天页面发送消息后，实际会调用：

- `/api/compat/assistant/chat/stream`

Next.js 再把这个请求代理到后端 compat API：

- `/compat/assistant/chat/stream`

后端流式聊天主逻辑集中在：

- `server/src/agent/services/compat_application_service.py`

虽然仓库里同时定义了标准的 LangGraph：

- `assistant_graph`
- `rag_query_graph`

但当前联调时，聊天流式主入口实际上优先走的是 compat service 中的手工编排逻辑。这样做的好处是：

- 更容易兼容 `assistant-ui` 的流式协议
- 更容易显式插入分析、检索、生成阶段事件
- 更容易把线程持久化和前端展示语义统一起来

## 3. 用户提问后的完整流程

## 3.1 前端发送消息

前端聊天运行时会把当前用户输入的最后一条文本消息提取出来，发送到：

```text
/api/compat/assistant/chat/stream
```

请求体里核心字段是：

```json
{
  "thread_id": "当前线程ID",
  "message": "用户本轮输入的问题"
}
```

这里的 `thread_id` 来自前端当前线程状态；如果没有现成线程，后端会自动创建。

## 3.2 后端先恢复线程

后端收到请求后，第一步不是立即检索知识库，而是先处理会话线程：

- 如果请求里有 `thread_id`
  - 先从 PostgreSQL 中读取这条线程的历史消息
- 如果没有 `thread_id`
  - 先创建一个新线程
  - 再把这个新线程作为本轮会话上下文

因此，这个系统并不是“无状态问答”，而是“带真实线程持久化的多轮问答”。

线程和消息主要落在 PostgreSQL 的两张表：

- `assistant_thread`
- `assistant_message`

## 3.3 把本轮问题接到历史消息后面

线程恢复后，后端会把本轮用户消息追加到历史消息列表中，形成一份新的消息序列：

- 历史消息
- 本轮用户消息

同时，系统还会提取最近最多 6 条用户/助手消息，整理成纯文本的 `conversation_context`，给后面的分析、记忆判断、生成使用。

也就是说，最近对话上下文本身会参与后续判断，而不是只有知识库参与。

## 3.4 先做“近期记忆能否直接回答”的判断

这是当前系统里一个很关键、也很容易被忽略的分支。

后端不会每次都直接走知识库，而是会先判断：

“这个问题能不能直接从最近对话里回答？”

比如：

- 用户刚刚说“我叫张三”
- 下一轮问“我叫什么名字”

这类问题没必要查知识库，直接用近期对话记忆即可。

这一步由 `ConversationService.answer_from_recent_history()` 完成，逻辑分两层：

- 优先使用 LLM 结合 `CONVERSATION_MEMORY_PROMPT` 判断
- 如果 LLM 不可用，再退回一些简单规则，比如从最近用户消息里抽名字

如果这一步返回了直接答案，那么后面的知识库召回流程会被跳过。

## 3.5 进入问题分析阶段

如果不能只靠近期记忆回答，系统就进入标准 RAG 的问题分析阶段。

这一阶段会产出一个结构化分析结果，主要字段有：

- `intent`
  - 问题意图，例如 `knowledge_qa`、`comparison`、`multi_hop`
- `need_rewrite`
  - 是否需要改写检索表达
- `need_rerank`
  - 是否需要重排
- `need_metadata_filter`
  - 是否需要元数据过滤
- `top_k`
  - 召回多少个候选 chunk

这一层优先使用：

- `DeepSeek`

具体是通过：

- `langchain_deepseek.ChatDeepSeek`

并用结构化输出直接解析为 `QueryAnalysisModel`。

如果大模型不可用，则会退回到启发式规则：

- 包含“区别 / 对比 / compare / difference”时，倾向认为是比较问题
- 包含“步骤 / how / why / 原因 / 流程”时，倾向认为是多跳问题
- 否则默认为普通知识问答

这一阶段的输出也会实时流给前端，前端会在当前用户消息下方显示：

- 问题分析中
- 问题分析完成
- 意图
- Top-K
- 是否改写
- 是否重排

## 3.6 查询改写

分析完成后，如果判断需要改写，就会进入查询改写阶段。

目标是把用户口语化的问题，改写成更适合向量检索的表达。

例如：

- 用户原问法偏口语、带上下文省略
- 改写后更完整、更适合作为检索 query

这一层同样优先使用 `DeepSeek`。

如果模型不可用，则退回到一个保底方案：

- 把问题做简单空白规整
- 至少保证检索字符串可用

最终得到一个：

- `rewritten_query`

后面的召回会优先使用这个改写后的检索表达。

## 3.7 向量召回

这是标准 RAG 里的“召回”步骤，当前项目里是有真正召回的，不是纯聊天直答。

### 召回使用的组件

- 向量模型：`BAAI/bge-m3`
- 向量库：`Milvus`
- LangChain 封装：`langchain_milvus.Milvus`

### 召回大致过程

1. 使用 `BGE-M3` 把检索 query 编码成向量
2. 在 `Milvus` 中按向量相似度搜索
3. 取回 `top_k` 个候选 chunk
4. 返回每个 chunk 的文本、分数和元数据

### 当前召回返回的 hit 结构

每条候选结果通常会包含：

- `chunk_id`
- `document_id`
- `content`
- `score`
- `metadata`

其中 `metadata` 里通常还会带：

- `title`
- `source_uri`
- 其他入库时带入的文档元数据

### 召回数量

召回数主要由两层决定：

- 默认配置：`retrieval_top_k`
- 分析阶段：`QueryAnalysisModel.top_k`

### 当前实现特征

当前召回本质上是：

- 向量召回为主
- 支持分数阈值 `score_threshold`
- 支持透传 `milvus_search_params`

但当前还没有真正把 `metadata filter` 下推到 Milvus 检索条件里，所以目前更准确地说是：

`纯向量召回 + 可选阈值 + 可选重排`

## 3.8 重排

召回出来的候选 chunk 不会直接全部喂给答案生成，而是会进入重排阶段。

### 重排使用的模型

- `BAAI/bge-reranker-v2-m3`

通过：

- `HuggingFaceCrossEncoder`

对每一对：

- `(用户问题, 候选 chunk 文本)`

进行打分，然后按打分排序。

### 为什么要重排

向量召回更像是“粗召回”，会先尽量多找相关候选；
重排则负责把最适合回答当前问题的片段排到前面。

### 如果重排模型不可用

系统不会崩，而是退回到一个词面重叠排序策略：

- 计算 query 和 chunk 文本的词面重合度
- 再结合原始召回分数进行排序

### 输出结果

最终保留前 `top_n` 条作为“最终上下文”，给后面的答案生成使用。

## 3.9 答案生成

完成召回和重排后，系统进入答案生成阶段。

### 输入上下文不是只有知识库

最终喂给生成模型的内容，实际上有两部分：

- 最近会话上下文
- 检索出来并重排后的知识片段

在代码里，生成前会把它们拼成两块：

- `[Recent Conversation]`
- `[Knowledge Base]`

也就是说，最终答案是基于：

`会话上下文 + 知识库上下文`

共同生成的。

### 生成模型

当前生成阶段优先使用：

- `DeepSeek`

也就是和分析、改写同一个 chat model。

### 流式输出

这一层不是等完整答案生成完才返回，而是：

- 每产出一段文本
- 就作为 `AIMessageChunk` 流给前端

所以前端聊天界面能看到真正的逐步出字，而不是最后整包回填。

### 如果模型不可用

系统会退回到一个骨架答案：

- 告诉你当前是离线 fallback
- 带上最近会话摘要
- 带上知识上下文摘要

这样至少能继续验证前后端链路和状态流，而不是整个系统直接不可用。

## 3.10 引用与轨迹组装

当答案生成完成后，后端不会只返回一段 answer 文本，还会把这一轮的辅助信息一起打包：

- `citations`
- `retrieval_hits`
- `turn_trace`
- reasoning 摘要

### citations

每条引用会包含：

- `document_id`
- `chunk_id`
- `title`
- `source_uri`

### retrieval_hits

会保留本轮参与召回/重排的候选摘要，包括：

- 文档标题
- 来源地址
- 片段得分
- snippet

### turn_trace

会按轮次保留三段轨迹：

- `analysis`
- `retrieval`
- `generation`

每一段都带状态和摘要信息。

这就是为什么前端不仅能显示答案，还能显示：

- 本轮分析了什么
- 检索了多少条片段
- 命中了哪些文档

## 3.11 持久化回 PostgreSQL

最终答案并不会只停留在前端内存里，而是会连同本轮用户消息一起保存回 PostgreSQL。

因此历史回放时，不仅能恢复消息正文，还能恢复：

- 引用信息
- 检索命中信息
- 单轮轨迹
- reasoning 摘要

这也是当前项目和很多简单 demo 的一个明显区别：

- 它不是“流式只在当前页面临时展示”
- 而是“实时体验和历史回放使用同一份持久化语义”

## 4. 当前使用的模型与组件一览

## 4.1 大模型

当前优先使用：

- `DeepSeek`

承担这些任务：

- 查询分析
- 查询改写
- 近期记忆判断中的 LLM 分支
- 最终答案生成

## 4.2 Embedding 模型

当前优先使用：

- `BAAI/bge-m3`

用途：

- 文档 chunk 向量化
- 查询向量化
- 向量召回

## 4.3 重排模型

当前优先使用：

- `BAAI/bge-reranker-v2-m3`

用途：

- 对召回结果做 cross-encoder 重排

## 4.4 向量库

当前使用：

- `Milvus`

用途：

- 存储 chunk 向量
- 执行相似度召回
- 支持 chunk 删除与重建

## 4.5 关系型数据库

当前使用：

- `PostgreSQL`

用途：

- 存线程与消息
- 存文档主数据
- 存文档版本
- 存入库任务和删除任务
- 存 chunk manifest
- 存源文件可重放信息

## 5. 一个直观的时序流程

```text
用户提问
  -> 前端发送 /compat/assistant/chat/stream
  -> 后端恢复线程或创建线程
  -> 提取最近 6 条消息形成会话上下文
  -> 判断是否可直接从近期会话回答
     -> 可以
        -> 直接生成答案
        -> 流式返回
        -> 持久化消息和轨迹
     -> 不可以
        -> DeepSeek 分析问题
        -> DeepSeek 改写检索表达
        -> BGE-M3 向量化 query
        -> Milvus 召回 Top-K chunk
        -> BGE-Reranker-v2-M3 重排
        -> DeepSeek 基于“会话上下文 + 知识上下文”生成答案
        -> 流式返回答案 chunk
        -> 写入 citations / retrieval_hits / turn_trace
        -> 持久化到 PostgreSQL
```

## 6. 和典型简单聊天系统的区别

当前项目相比一个普通聊天 demo，主要多了这些工程化能力：

- 有真实线程持久化，而不是纯前端内存会话
- 有近期记忆判断，而不是每轮都强制查库
- 有查询分析和查询改写
- 有真正的 Milvus 向量召回
- 有可选重排，不是召回结果直接生成
- 有流式阶段事件，不只是最终答案
- 有历史轨迹回放，而不是运行中看到、刷新后消失

## 7. 当前已具备但仍未完全做深的点

从当前代码看，以下能力已经有接入位，但还不是最终生产态：

- `metadata filter`
  - 状态模型里已经有字段，但检索阶段还没有真正下推过滤条件
- 多知识库路由
  - 图里有 `route_query` 节点，但当前更多是保留扩展位
- 更完整的评测与观测
  - 当前更偏“第一阶段可运行工程骨架”
- 更成熟的检索策略
  - 例如 hybrid retrieval、query router、self-RAG 等还未真正落地

## 8. 总结

如果用一句更工程化的话总结当前问答主链路，可以描述为：

> 这是一个以 `DeepSeek` 负责分析/改写/生成、以 `BGE-M3 + Milvus` 负责向量召回、以 `BGE-Reranker-v2-M3` 负责候选重排、以 `PostgreSQL` 负责线程与轨迹持久化的流式多轮 RAG 问答系统。

如果继续往下拆，它的真实执行顺序就是：

`恢复线程 -> 看最近对话能不能直接答 -> 不能则分析 -> 改写 -> 召回 -> 重排 -> 生成 -> 回写轨迹`

这就是当前项目聊天问答能力的完整精细化处理流程。
