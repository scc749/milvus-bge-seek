# 后端架构规范文档

## 1. 目标

本后端以 `LangGraph` 为核心编排层，以“API 适配层 / 图编排层 / 应用服务层 / 领域服务层 / 仓储层 / 基础设施层”分层组织，目标是：

- 图负责流程编排，不承担基础设施细节。
- 应用服务负责 use case，不直接暴露给前端 HTTP。
- 仓储层只负责持久化与查询，不混入页面语义。
- 失败语义真实，不允许静默假成功。

## 2. 目标目录结构

```text
server/src/agent/
├─ api/                        # FastAPI 适配层
│  ├─ compat_models.py
│  └─ compat_router.py
├─ nodes/                      # LangGraph 节点
├─ services/                   # 应用服务 / 领域服务
├─ repositories/               # PostgreSQL 仓储
├─ schemas/                    # 状态、常量、契约
├─ components/                 # provider / factory
├─ graph.py                    # 图定义与图装配
├─ dependencies.py             # 依赖容器
└─ compat_api.py               # FastAPI app 入口
```

## 3. 分层职责

### 3.1 API 适配层 `api/`

职责：

- 定义 HTTP 路由。
- 做请求模型校验。
- 调用应用服务。
- 将 HTTP 错误转换为合适响应。

禁止：

- 在路由中直接拼接复杂业务流程。
- 在路由中直接操作数据库或向量库。
- 在路由中维护大段线程/状态逻辑。

### 3.2 图编排层 `graph.py + nodes/`

职责：

- 描述 LangGraph 的步骤顺序。
- 通过节点协调应用服务。
- 保持状态字段清晰、最小、可追踪。

要求：

- 节点应尽量薄，只做一步动作。
- 节点命名必须体现业务动作，而不是技术细节。

### 3.3 应用服务层 `services/*application*`

职责：

- 承载具体 use case。
- 组合多个领域服务、仓储与图结果。
- 输出前端或图节点可消费的稳定结果结构。

当前应重点包含：

- `AdminApplicationService`
- `CompatApplicationService`
- `IngestApplicationService`
- `DeleteApplicationService`
- `ReindexApplicationService`

### 3.4 领域服务层 `services/`

职责：

- 封装单一领域能力。
- 例如：
  - `ConversationService`
  - `IngestionService`
  - `DocumentSourceStoreService`
  - `MilvusAdminService`

禁止：

- 一个服务同时承担多个 use case 编排职责。
- 将返回对象设计成页面专用 DTO。

### 3.5 仓储层 `repositories/`

职责：

- PostgreSQL 读写。
- 明确事务边界。
- 不感知前端页面契约。

要求：

- 写模型仓储必须显式事务。
- 读模型仓储不得把“依赖失败”伪装成“空数据”。

### 3.6 向量基础设施约束

要求：

- `langchain_milvus.Milvus` 相关接入必须与 `pymilvus` 底层连接语义保持一致。
- 如果同时使用 `MilvusClient` 和 ORM `Collection`，必须确保 ORM alias 已按同一连接配置注册。
- 向量库连接异常可以返回业务失败态，但不能让 HTTP 接口退化为非结构化 500 文本。

## 4. 依赖方向

允许：

- `api -> services`
- `nodes -> services`
- `services -> repositories`
- `services -> components`
- `repositories -> config / schema manager`

禁止：

- `repositories -> services`
- `services -> api`
- `graph -> api`
- `nodes -> HTTP request model`

## 5. 图设计规范

### 5.1 查询图

- `assistant_graph` 是消息态唯一入口。
- `rag_query_graph` 是纯查询态内部流程。
- 对接 `assistant-ui` 时，应优先走 `assistant_graph`，不要在 HTTP 入口旁路消息态适配。

### 5.2 入库 / 删除 / 重建图

必须保证：

- 任务状态真实。
- Milvus 写入失败时，任务必须 `failed`。
- 删除失败时，任务必须 `failed`。
- 回放来源优先使用受控 `replay_source_uri`。
- `ingest` 与 `reindex` 作为任务型动作时，HTTP 接口默认语义应为“先创建任务并立即返回 `job_id`，后端后台继续执行图”。
- 禁止将“任务提交成功”混同为“任务执行完成”，否则前端会出现长时间阻塞和错误的加载语义。
- URL 入库若启用递归抓取，必须限制在当前 URL 范围内，并把根 URL 与页面 URL 分开建模：任务入口保留根 URL，页面文档保留真实子页面 URL。
- 递归 URL 抓取的子页面不得在持久化时被统一覆盖成根 URL；至少要保留根 URL、页面 URL、页面路由与抓取模式等元数据。

## 6. compat API 规范

compat API 的定位是：

- 在 Python 3.10 环境下，作为官方 LangGraph Server 的本地兼容桥接层。
- 服务于 `assistant-ui` 和管理台的本地联调。

设计原则：

- `compat_api.py` 只保留 app 入口。
- `compat_router.py` 只保留路由。
- `CompatApplicationService` 负责线程管理、图调用与返回结构归一。
- 助手聊天允许提供专门的流式 compat 端点，但必须保持统一线程与持久化语义。
- 流式 compat 事件可以显式输出分析、检索、生成阶段，但这些阶段数据最终必须沉淀到助手消息元数据中，确保历史回放与实时体验使用同一套契约。
- 若前端运行时基于 `assistant-ui/useLangGraphRuntime`，则 `messages` 流式事件必须遵循 LangGraph tuple 结构 `[message, metadata]`，同一轮 `AIMessageChunk` 必须使用稳定 `id`，否则前端会退化为仅在最终 `values` 阶段整包回填。

## 7. 状态与契约规范

### 7.1 LangGraph State

要求：

- 只保留流程真正需要的字段。
- 避免重复字段与歧义字段。
- 结果字段应通过 `result` 或明确命名输出，不要混杂临时变量与最终返回。

### 7.2 Page Contract

要求：

- 如果后端定义 page contract，前端必须明确消费哪些部分。
- 不允许后端维护一整套复杂页面描述，而前端完全不用。
- 列表页 contract 至少要显式输出可消费的 `tables`。
- 详情页 contract 至少要显式输出 `sections.fields`、`sections.stat_cards`、`sections.tables` 之一，禁止只保留模糊说明文本。

建议：

- 当前阶段优先把 contract 收敛为：
  - query schema
  - filter schema
  - sort schema
  - table/detail contract

### 7.3 Admin 查询分发

要求：

- `AdminApplicationService` 不再使用持续膨胀的 `if/elif` 字符串分发。
- 每个 `operation` 必须注册到显式 handler map，由应用服务统一解析分页、排序、筛选上下文后再调用 repository 或 contract service。
- 未注册 operation 必须返回显式错误与 `available_operations`，不能静默回退到默认查询。

### 7.4 聊天线程持久化

要求：

- 聊天线程历史不能只保存在内存。
- 至少需要持久化：
  - thread 元数据
  - 有序消息列表
  - 助手消息附带的 reasoning / citations / turn_trace 等补充元数据
- 聊天流式接口即使失败，也要优先返回结构化错误或业务失败态，不能退化为不可解析的纯文本 500。
- `turn_trace` 应至少覆盖：
  - `analysis`
  - `retrieval`
  - `generation`
  并能表达运行状态、检索摘要与检索命中列表。
- 前端实时展示允许先消费流式阶段事件挂在当前用户问题下方，但最终仍必须以持久化后的助手消息元数据为准完成历史回放。
- 若前端消息运行时基于 `assistant-ui` 的 LangGraph 转换器，则助手消息的自定义持久化元数据不能只放在 `response_metadata`，还必须镜像到 `additional_kwargs.metadata.custom`，否则当前会话结束后整包替换消息时会丢失这些前端自定义展示信息。

## 8. 命名规范

- 图：`assistant_graph`、`rag_ingest_graph`
- 节点：`load_documents`、`finalize_ingestion`
- 应用服务：`CompatApplicationService`
- 领域服务：`ConversationService`
- 仓储：`PostgresReadRepository`

禁止：

- 模糊命名，例如 `process()`, `handle()`, `do_task()`

## 9. 错误处理规范

- 基础设施错误应显式暴露给应用层。
- 应用层决定是否降级。
- 仓储层不要无差别 `except Exception: return []`
- 图的终态必须带明确 `status/error`

## 10. 当前阶段重构结论

本次已完成的第一阶段收敛：

- 将 compat API 拆为 `app / router / models / application service`
- compat 聊天链路切回消息态 `assistant_graph`
- 引入 `CompatApplicationService` 统一线程与图调用边界

本次已完成的第二阶段收敛：

- 删除 `DocumentLifecycleService` 这类大一统生命周期服务
- 将入库、删除、重建拆为 `IngestApplicationService` / `DeleteApplicationService` / `ReindexApplicationService`
- 引入 `DocumentSourceService`，收口 source metadata enrichment、public metadata 清洗与重建 identity 固定逻辑
- LangGraph 节点直接依赖具体 use case，而不是再穿过大而全的生命周期门面

本次已完成的第三阶段收敛：

- 删除 `PostgresRegistryService` / `PostgresReadModelService` 两层薄封装
- 应用服务直接依赖 `PostgresRegistryRepository` / `PostgresReadRepository`
- `read model` 与 `write model` 的边界下沉到 repository 层，应用服务只负责 use case 编排
- 管理查询入口继续保留在 `AdminApplicationService`，但其底层读取已经直接走 `PostgresReadRepository`

后续建议继续推进：

- 进一步收敛 read-model 查询异常语义，避免静默降级
- 继续为 ingest/delete/reindex use case 增补更完整的单测覆盖

本次已完成的第四阶段收敛：

- `AdminApplicationService` 已收敛为显式 operation registry + 统一 `AdminQueryContext`
- `document_list / ingest_job / delete_job` contract 已显式输出列表表格 `tables`
- 详情类 contract 的 `sections/stat_cards/tables` 已作为前端真实消费边界，不再只做“部分消费”

本次已完成的聊天链路收敛：

- `compat assistant` 已补充线程列表、线程删除、线程更新与流式聊天端点
- 聊天线程已持久化到 PostgreSQL `assistant_thread / assistant_message`
- 流式聊天已显式输出分析、检索、生成阶段事件，并回写最终消息的 `reasoning / citations / turn_trace`
- 聊天前端不再依赖页面顶部固定步骤面板，而是消费助手消息元数据恢复每轮会话轨迹
