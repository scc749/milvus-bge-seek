# 前端对接清单

本文档面向后续 `client` 文档中心与任务中心页面开发，约定每个页面应该调用的 `rag_admin_graph` operation、输入参数和主要渲染字段。

当前新增约束：

- 列表页优先消费 contract 中的 `tables`，不要在前端重复维护列清单。
- 详情页优先消费 contract 中的 `sections.fields`、`sections.stat_cards`、`sections.tables`，不要只消费 `filters/sorts`。

当前本地联调调用路径：

- 浏览器访问 `client`
- `client` 通过 Next.js 代理请求 `/api/compat/*`
- 代理再转发到后端 `FastAPI compat API`
- `compat_api.py` 内部再调用本地 `rag_admin_graph` / `rag_ingest_graph` / `rag_delete_graph` / `rag_reindex_graph`

当前已验证可用的接口：

- `POST /compat/admin/page-contract`
- `POST /compat/admin/query`
- `POST /compat/ingest`
- `POST /compat/delete`
- `POST /compat/reindex`
- `POST /compat/assistant/threads`
- `POST /compat/assistant/chat`

## 1. 文档列表页

页面目标：

- 展示文档列表
- 支持分页、筛选、搜索、排序
- 支持跳转到文档详情页

页面契约：

- 先调用：
  - `{"operation": "get_page_contract", "page_name": "document_list"}`
- 再调用主查询：
  - `{"operation": "list_documents", "page": 1, "page_size": 20}`

支持参数：

- `page`
- `page_size`
- `status_filter`
- `source_type_filter`
- `query`
- `sort_by`
- `sort_direction`

列表渲染字段：

- `title`
- `document_id`
- `status`
- `source_type`
- `current_version_number`
- `current_chunk_count`
- `updated_at`

当前前端已按 `document_table` 直接渲染上述列结构。

推荐交互：

- 点击列表项后，使用 `document_id` 跳转到文档详情页

## 2. 文档详情页

页面目标：

- 展示文档主信息
- 展示版本/chunk 统计
- 展示最近版本和完整版本历史

页面契约：

- 先调用：
  - `{"operation": "get_page_contract", "page_name": "document_detail"}`
- 主查询：
  - `{"operation": "get_document_detail", "document_id": "doc-123"}`
- 关联查询：
  - `{"operation": "list_document_versions", "document_id": "doc-123", "page": 1, "page_size": 20}`

主记录渲染字段：

- `title`
- `document_id`
- `status`
- `source_type`
- `source_uri`
- `replay_source_uri`
- `source_storage.storage_uri`
- `updated_at`

说明：

- `source_uri` 是用户可理解的原始来源
- `replay_source_uri` 和 `source_storage.*` 来自后端独立表 `document_source_storage`

统计卡字段：

- `stats.version_total`
- `stats.completed_versions`
- `stats.active_chunk_total`
- `stats.superseded_versions`

最近版本渲染字段：

- `recent_versions[].version_number`
- `recent_versions[].status`
- `recent_versions[].chunk_count`
- `recent_versions[].updated_at`

版本历史表字段：

- `version_number`
- `status`
- `content_hash`
- `chunk_count`
- `updated_at`

推荐布局：

- `overview` Tab
  - `summary`
  - `stats`
- `versions` Tab
  - `recent_versions`
  - `versions_table`

当前前端已按 `summary / stats / recent_versions / versions_table` contract 真实消费。

## 5. 对接建议

管理页面建议统一流程：

1. 首次进入页面，先调用 `get_page_contract`
2. 按 contract 动态初始化过滤项、排序项、tabs 和 sections
   - 列表优先读 `tables`
   - 详情优先读 `sections.fields / sections.stat_cards / sections.tables`
3. 再调用页面主查询 operation
4. 详情页按 `secondary_operations` 拉取关联数据
5. 文档入库调用 `rag_ingest` 时，优先使用以下两种输入之一：
   - `{"source_uri":"E:\\docs\\manual.pdf","backup_source":true}`
   - `{"source_name":"manual.pdf","source_mime_type":"application/pdf","source_content_b64":"<base64>","backup_source":true}`

这样做的好处是：

- 前端和后端共享同一套页面约束
- 后续改字段或增加 section 时，不必两端各自猜测
- 便于逐步把文档中心和任务中心从静态页面演进为配置驱动页面
